# Benchmark Reproducibility Contract

## Purpose

Bench numbers in this codebase have historically drifted because each bench used different prompts, thermal state, KV type, or even different binary builds (NDEBUG vs not). This document fixes the protocol so two engineers running the same bench on the same hardware get matching results within noise. Bench numbers that violate this contract are not citable in PRs or commit messages.

## Hardware Fingerprint (Required at Run Head)

The bench script must capture and print the following before any measurement begins:

- **GPU model + driver version** — run `nvidia-smi --query-gpu=name,driver_version --format=csv`
- **GPU power limit** — must be explicitly set before the run (e.g., `nvidia-smi -pl 350` for RTX 3090). Ambient power scaling invalidates all comparisons.
- **CUDA toolkit version** — `nvcc --version`
- **Commit SHA** of the binary under test — `git rev-parse HEAD`
- **KV quantization type** — q8_0, q4_0, f16, TQ3_0, or other
- **CMake build type** — must be `Release`. Debug builds skew everything.
- **NDEBUG confirmation** — run a probe that would trigger an assertion if NDEBUG is not set. If asserts fire during the bench, that's a build config bug, not a bench result. Discard the run.

All fingerprint data must go into the bench output JSON (see Bench Output Format below), not just console logs. Fingerprints are immutable per run.

## Paired Interleaved A/B Methodology

The only acceptable shape for A/B performance comparisons:

**Family constraints:**
- Same family vs same family: Dense ↔ Dense, MoE ↔ MoE. Never cross-family comparisons. MoE active params are a fraction of total — comparing active MoE params to dense total params is architecturally meaningless and confuses speedup claims.

**Prompt and run structure:**
- Identical prompt set for both A and B (n ≥ 5 prompts, HumanEval-style or matched distribution)
- Interleaved measurement order: A B A B A B … (not A A A then B B B)
  - Interleaving ensures thermal drift hits both arms equally
- Minimum 3 runs per cell, report median tok/s (not mean — median is outlier-robust)
- If GPU temp exceeds the configured threshold during a single run, discard that run and rerun cold

**Thermal management:**
- Capture GPU temperature for the full bench duration
- Record the thermal log in the output JSON
- If any run exceeds the threshold, discard and rerun

## Forbidden Bench Shapes

These outputs are not acceptable as evidence in PRs or commit messages:

- One prompt, one run, one number — never acceptable
- A and B measured on different days, different driver versions, or different power limits
- Mean tok/s without sample count N and standard deviation (or stddev σ in a table)
- Bench results from a binary built without `-DCMAKE_BUILD_TYPE=Release`
- "Beats baseline" claims without the baseline number shown in the same table
- Hand-edited bench tables in PR descriptions (must be generated by the bench script, not written by hand)
- Comparisons across different KV types (q8_0 vs q4_0) without explicit acknowledgment that quantization load varies

## Bench Output Format

Bench scripts must output two artifacts:

### JSON Output

The primary artifact, stored under `dflash/bench/results/<date>-<topic>.json`:

```json
{
  "timestamp": "2026-05-15T14:32:00Z",
  "commit": "abc1234def567890",
  "gpu": {
    "model": "RTX 3090",
    "driver_version": "550.120"
  },
  "cuda_version": "12.4",
  "power_limit_watts": 350,
  "kv_type": "q8_0",
  "build_type": "Release",
  "ndebug_verified": true,
  "prompts": [
    {"id": 1, "text": "Implement mergesort...", "tokens": 42},
    {"id": 2, "text": "Explain OAuth...", "tokens": 38}
  ],
  "cells": [
    {
      "name": "27B Dense AR baseline",
      "tok_s_median": 31.42,
      "tok_s_stddev": 0.18,
      "runs": [31.55, 31.18, 31.44],
      "gpu_temp_max_celsius": 82
    },
    {
      "name": "27B Dense + Qwen3.6 MTP γ=2",
      "tok_s_median": 47.81,
      "tok_s_stddev": 0.31,
      "runs": [47.62, 48.05, 47.76],
      "gpu_temp_max_celsius": 83,
      "speedup_vs_baseline": 1.52
    }
  ]
}
```

### Markdown Table (Derived View)

Include in PR descriptions or commit messages as a human-readable summary. This must be generated from the JSON, not hand-edited:

```
RTX 3090 @ 350W, KV q8_0, commit abc1234, NDEBUG verified, n_predict=128, 5 HE prompts × 3 runs median:

| Configuration                    | tok/s    | σ      | Speedup |
|----------------------------------|----------|--------|---------|
| 27B Dense AR baseline            | 31.42    | 0.18   | —       |
| 27B Dense + Qwen3.6 MTP γ=2      | 47.81    | 0.31   | 1.52×   |
| 35B-A3B AR baseline              | 28.05    | 0.22   | —       |
| 35B-A3B + Qwen3.6 MTP γ=2        | 41.93    | 0.27   | 1.49×   |
```

Speedup is calculated as: `tok_s_new / tok_s_baseline`, rounded to two decimals.

## When to Bench

- Before opening any PR that claims a speedup (required, not optional)
- Before claiming a regression-free refactor (must show ±2% match against the last green bench)
- After any kernel change or graph-builder change that affects inference latency
- When adding support for a new model, quantization type, or inference feature

Bench numbers in planning documents (e.g., gemma4_pr_split_plan.md) start as placeholders (`??.??`) and must be filled by the bench script during validation, not by hand-editing.
